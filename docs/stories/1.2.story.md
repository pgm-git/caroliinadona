## Descricao

Configurar o Supabase como backend (auth, database, storage). Criar clients para server e browser, configurar variaveis de ambiente e preparar para multi-tenancy. Esta story estabelece toda a camada de comunicacao entre o frontend Next.js e o backend Supabase, incluindo o middleware de refresh de sessao que e critico para a autenticacao funcionar corretamente com Server Components.

## Notas de Implementacao

### Browser Client (`client.ts`)

O browser client deve ser criado usando `createBrowserClient` do `@supabase/ssr`. Este client e usado em Client Components e hooks:

```typescript
import { createBrowserClient } from '@supabase/ssr'

export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
}
```

### Server Client (`server.ts`)

O server client deve ser criado usando `createServerClient` do `@supabase/ssr` com acesso a cookies do Next.js. Este client e usado em Server Components, Server Actions e Route Handlers:

```typescript
import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export async function createClient() {
  const cookieStore = await cookies()
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll() },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {
            // The `setAll` method was called from a Server Component.
            // This can be ignored if you have middleware refreshing sessions.
          }
        },
      },
    }
  )
}
```

### Admin Client (`admin.ts`)

O admin client usa a `SUPABASE_SERVICE_ROLE_KEY` e tem acesso total ao banco, ignorando RLS. NUNCA deve ser importado em client components. Usar apenas em:
- Server Actions que precisam de acesso admin
- API Routes internas
- Scripts de migracao/seed

```typescript
import { createClient } from '@supabase/supabase-js'

export const adminClient = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
  { auth: { autoRefreshToken: false, persistSession: false } }
)
```

### Middleware (`middleware.ts` e `supabase/middleware.ts`)

O middleware do Next.js deve chamar a funcao de refresh do Supabase em toda request para manter a sessao ativa. O `src/lib/supabase/middleware.ts` contem a logica de refresh, e o `src/middleware.ts` e o ponto de entrada do Next.js que invoca essa logica.

O middleware deve:
1. Criar um server client com acesso a cookies da request
2. Chamar `supabase.auth.getUser()` para forcar o refresh do token
3. Repassar os cookies atualizados na response

Configurar o `matcher` do middleware para ignorar arquivos estaticos:
```typescript
export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
}
```

### Variaveis de Ambiente

O arquivo `.env.local.example` deve conter:

```env
# Supabase - Public (expostas no browser)
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key-here

# Supabase - Private (apenas server-side)
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key-here
SUPABASE_DB_URL=postgresql://postgres:postgres@127.0.0.1:54322/postgres

# App
NEXT_PUBLIC_APP_URL=http://localhost:3000
```

### Validacao de Env (`env.ts`)

Criar um modulo de validacao que verifica no startup se todas as variaveis obrigatorias estao definidas. Usar uma abordagem simples sem dependencias externas (sem zod aqui, sera adicionado na story de tRPC):

```typescript
function getRequiredEnv(key: string): string {
  const value = process.env[key]
  if (!value) {
    throw new Error(`Missing required environment variable: ${key}`)
  }
  return value
}

export const env = {
  supabase: {
    url: getRequiredEnv('NEXT_PUBLIC_SUPABASE_URL'),
    anonKey: getRequiredEnv('NEXT_PUBLIC_SUPABASE_ANON_KEY'),
  },
} as const
```

As variaveis `SUPABASE_SERVICE_ROLE_KEY` e `SUPABASE_DB_URL` devem ser validadas apenas quando acessadas (lazy validation), pois nao sao necessarias no browser.

### Supabase Local Development

O `supabase/config.toml` ja deve existir (criado pelo `supabase init`). Ajustar configuracoes relevantes:
- `[auth]` site_url para `http://localhost:3000`
- `[auth]` additional_redirect_urls para incluir `http://localhost:3000/**`
- `[api]` schemas para incluir `public` e `auth`
