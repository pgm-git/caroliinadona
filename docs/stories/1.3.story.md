1. **00001** - Tipos ENUM base (case_status, case_type, document_type, etc.)

3. **00003** - Tabela profiles (extensao da auth.users com dados de perfil)
4. **00004** - Tabela cases (entidade central do sistema)
5. **00005** - Tabela case_subjects (partes do processo: autor, reu, etc.)
6. **00006** - Tabela documents (documentos anexados aos casos)

### ENUMs Esperados (12)

- `case_status` - Status do caso (draft, pending_documents, documents_uploaded, extracting, extracted, validating, validated, calculating, calculated, generating_petition, petition_generated, filed, archived)
- `case_type` - Tipo de acao (execucao_titulo_extrajudicial, monitoria, busca_apreensao, etc.)
- `document_type` - Tipo de documento (contrato, extrato, notificacao, procuracao, etc.)
- `document_status` - Status do documento (uploaded, processing, processed, error)
- `calculation_type` - Tipo de calculo (correcao_monetaria, juros_mora, multa, honorarios)
- `calculation_index` - Indice de correcao (igpm, ipca, inpc, selic, cdi)
- `petition_status` - Status da peticao (draft, review, approved, filed)
- `user_role` - Papel do usuario (admin, lawyer, assistant, viewer)
- `subject_type` - Tipo de parte (autor, reu, fiador, avalista, etc.)
- `notification_type` - Tipo de notificacao (info, warning, error, success)
- `audit_action` - Acao de auditoria (INSERT, UPDATE, DELETE)
- `organization_plan` - Plano da organizacao (trial, starter, professional, enterprise)

### RLS Policies

Todas as tabelas usam RLS baseado na `organization_id` do usuario autenticado. O padrao e:
- O usuario so ve/edita dados da sua propria organizacao
- A `organization_id` e extraida dos claims do JWT via `auth.jwt() ->> 'organization_id'`
- Policies de SELECT, INSERT, UPDATE e DELETE sao criadas separadamente

### Triggers de Audit Log

7 tabelas possuem triggers que registram alteracoes no `audit_log`:
- `cases` (INSERT, UPDATE, DELETE)
- `case_subjects` (INSERT, UPDATE, DELETE)
- `documents` (INSERT, UPDATE, DELETE)
- `calculations` (INSERT, UPDATE, DELETE)
- `petitions` (INSERT, UPDATE, DELETE)
- `profiles` (UPDATE, DELETE)
- `organizations` (UPDATE, DELETE)

O trigger chama uma funcao `log_audit_event()` que registra:
- `table_name` - nome da tabela
- `record_id` - UUID do registro afetado
- `action` - INSERT/UPDATE/DELETE
- `old_data` - dados anteriores (para UPDATE/DELETE)
- `new_data` - dados novos (para INSERT/UPDATE)
- `user_id` - usuario que fez a alteracao
- `organization_id` - organizacao do usuario

### Verificacao Pos-Aplicacao

Apos aplicar as migrations, executar queries de verificacao:

```sql
-- Contar tabelas
SELECT count(*) FROM information_schema.tables
WHERE table_schema = 'public' AND table_type = 'BASE TABLE';

-- Contar ENUMs
SELECT count(DISTINCT t.typname)
FROM pg_type t JOIN pg_namespace n ON t.typnamespace = n.oid
WHERE n.nspname = 'public' AND t.typtype = 'e';

-- Verificar RLS habilitado
SELECT tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public';

-- Contar RLS policies
SELECT count(*) FROM pg_policies WHERE schemaname = 'public';

-- Verificar triggers
SELECT trigger_name, event_object_table, event_manipulation
FROM information_schema.triggers
WHERE trigger_schema = 'public';

-- Verificar seed data
SELECT count(*) FROM comarcas;
SELECT count(*) FROM petition_templates;
```

### Troubleshooting

Se alguma migration falhar:
1. Verificar logs com `supabase db reset` para comecar do zero
2. Checar dependencias entre migrations (ex: ENUM deve existir antes de ser usado em CREATE TABLE)
3. Verificar se funcoes referenciadas existem (ex: `log_audit_event()` deve existir antes dos triggers)
4. Para erros de permissao, verificar se o schema `public` tem as grants corretas

### Full-Text Search

A migration 00013 deve configurar full-text search na tabela `documents`:
- Coluna `search_vector` do tipo `tsvector`
- Trigger para atualizar automaticamente o `search_vector` em INSERT/UPDATE
- Index GIN na coluna `search_vector` para buscas performaticas
- A funcao `search_cases` RPC deve usar esse index para busca
