## Descricao

Configurar Drizzle ORM como camada de acesso ao banco de dados. Gerar schema TypeScript a partir das migrations existentes, criar client de conexão, configurar tipos inferidos e preparar helpers de query que respeitam multi-tenancy (org_id).

- [ ] Drizzle ORM instalado (`drizzle-orm`, `drizzle-kit`, `postgres`)
- [ ] Schema TypeScript gerado refletindo todas as 15 tabelas do banco
- [ ] 12 ENUMs mapeados como `pgEnum` no Drizzle
- [ ] Client de conexão configurado (`src/server/db/client.ts`)
- [ ] `drizzle.config.ts` configurado para introspecção do Supabase
- [ ] Tipos inferidos exportados para cada tabela (`$inferSelect`, `$inferInsert`)
- [ ] Helper `withOrgId(query, orgId)` que adiciona filtro de tenant automaticamente
- [ ] Relations definidas entre tabelas (organization → cases → documents → extracted_data, etc.)
- [ ] Drizzle Studio funcional (`npx drizzle-kit studio`)
- [ ] Queries tipadas funcionando end-to-end

## Notas de Implementacao

- O schema Drizzle deve refletir EXATAMENTE as migrations SQL em `supabase/migrations/`
- Usar `pgTable`, `pgEnum`, `uuid`, `text`, `timestamp`, `numeric`, `boolean`, `jsonb`, `integer` do `drizzle-orm/pg-core`
- Usar driver `postgres` (not `pg`) para conexão — mais moderno e performático
- Connection string vem de `DATABASE_URL` no `.env.local`
- O helper `withOrgId` deve ser usado em TODA query que acessa dados multi-tenant para garantir isolamento
- Relations servem para eager loading com `db.query.cases.findMany({ with: { documents: true, parties: true } })`
- ENUMs devem usar os mesmos nomes das migrations SQL
- Tabelas a mapear: organizations, users, org_members, courts, cases, parties, documents, extracted_data, validations, calculations, status_history, petition_templates, petitions, exceptions, audit_log
