## Descricao

Implementar a camada de integracao com Supabase Storage para upload, download e gerenciamento de documentos. Definir a estrutura padronizada de paths, criar o bucket `case-documents` com policies de acesso baseadas em RLS, e implementar o service layer que o componente de upload (2.1) consumira. Cada documento deve ser armazenado com path padronizado: `{org_id}/{case_id}/{document_id}.{ext}`.

## Notas de Implementacao

### Estrutura de Paths no Storage

```
case-documents/
  {org_id}/
    {case_id}/
      {document_id}.pdf
      {document_id}.jpg
      {document_id}.png
```

O `document_id` e gerado como UUID antes do upload, garantindo unicidade. A extensao original e preservada.

### Migration para Storage Policies

```sql
-- Criar bucket (se nao existir)
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'case-documents',
  'case-documents',
  false,
  26214400, -- 25 MB
  ARRAY['application/pdf', 'image/jpeg', 'image/png', 'image/tiff']
)
ON CONFLICT (id) DO NOTHING;

-- Policy: usuarios autenticados podem fazer upload na pasta da sua org
CREATE POLICY "org_upload" ON storage.objects
  FOR INSERT
  WITH CHECK (
    bucket_id = 'case-documents'
    AND (storage.foldername(name))[1] = (auth.jwt() ->> 'organization_id')
  );

-- Policy: usuarios autenticados podem ler documentos da sua org
CREATE POLICY "org_read" ON storage.objects
  FOR SELECT
  USING (
    bucket_id = 'case-documents'
    AND (storage.foldername(name))[1] = (auth.jwt() ->> 'organization_id')
  );

-- Policy: usuarios autenticados podem deletar documentos da sua org
CREATE POLICY "org_delete" ON storage.objects
  FOR DELETE
  USING (
    bucket_id = 'case-documents'
    AND (storage.foldername(name))[1] = (auth.jwt() ->> 'organization_id')
  );
```

### Storage Service

```typescript
interface UploadDocumentInput {
  orgId: string;
  caseId: string;
  file: File | Buffer;
  originalFilename: string;
  mimeType: string;
  documentType: DocumentType;
  uploadedBy: string;
}

interface UploadDocumentResult {
  documentId: string;
  storagePath: string;
  fileHash: string;
}

class StorageService {
  async uploadDocument(input: UploadDocumentInput): Promise<UploadDocumentResult>;
  async getSignedUrl(storagePath: string, expiresIn?: number): Promise<string>;
  async deleteDocument(documentId: string): Promise<void>;
}
```

### Hash SHA-256

O hash do arquivo deve ser calculado server-side usando `crypto.createHash('sha256')`. O hash e armazenado na coluna `file_hash` da tabela `documents` e pode ser usado futuramente para deteccao de duplicatas (FR-007, v1.5).

### Validacao Server-Side

Mesmo que o cliente valide, o servidor DEVE revalidar:
- Tamanho maximo: 25 MB (26214400 bytes)
- MIME types aceitos: application/pdf, image/jpeg, image/png, image/tiff
- Arquivo nao vazio (size > 0)

### Tabela Documents (ja existe no schema)

A tabela `documents` ja foi criada na story 1.7 (Drizzle ORM). Campos relevantes:
- `storage_bucket`: "case-documents"
- `storage_path`: `{org_id}/{case_id}/{document_id}.{ext}`
- `original_filename`: nome original do arquivo
- `mime_type`: tipo MIME
- `file_size_bytes`: tamanho em bytes
- `file_hash`: SHA-256
- `uploaded_by`: UUID do usuario que fez upload
- `metadata`: JSON com canal de entrada, cooperativa, lote, etc.
