## Descricao

Implementar Role-Based Access Control (RBAC) com 4 roles (admin, coordinator, lawyer, intern). Criar middleware de autorização que valida permissões por rota e por operação, integrado com RLS do Supabase para isolamento multi-tenant.

## Criterios de Aceite

- [ ] 4 roles definidos: admin (full access), coordinator (manage team + cases), lawyer (manage own cases), intern (view + upload only)
- [ ] Middleware de autorização valida role do usuário antes de permitir acesso
- [ ] Permissões granulares por módulo (cases, documents, petitions, templates, settings, users)
- [ ] Helper `can(user, action, resource)` reutilizável em server e client
- [ ] Admin pode gerenciar usuários do escritório (CRUD de membros)
- [ ] Coordinator pode atribuir casos e ver todos os casos do escritório
- [ ] Lawyer pode criar/editar/ver apenas seus casos atribuídos
- [ ] Intern pode fazer upload e ver casos, mas não editar petições
- [ ] RLS ativo: usuário de org A não acessa dados de org B (já configurado via migrations)
- [ ] Página de gerenciamento de usuários (`/settings/team`) para admin
- [ ] Convite de novos membros por email (admin only)

## Notas de Implementacao

- Mapa de permissões segue o padrão: `{ [role]: { [resource]: ['create', 'read', 'update', 'delete'] } }`
- Permissões por role:
  - **admin:** CRUD em tudo (cases, documents, petitions, templates, settings, users)
  - **coordinator:** CRUD em cases/documents/petitions, read templates, read settings, read users
  - **lawyer:** CRU em own cases/documents, read petitions, no access settings/users
  - **intern:** R em cases, CR em documents (upload), no access petitions/templates/settings/users
- RLS garante isolamento por org_id no banco — RBAC garante autorização por role na aplicação
- O middleware tRPC deve ser composável: `protectedProcedure.use(requireRole('admin'))`
- Usar Supabase Auth admin API (service role) para convidar membros
