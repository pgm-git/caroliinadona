## Descricao

Configurar BullMQ como sistema de filas assíncronas com Redis (Upstash) como backend. Criar as filas principais do sistema, workers placeholder, health check de conexão e dashboard de monitoramento (BullBoard). Essencial para o processamento assíncrono de documentos (OCR, extração, geração de petições).

## Criterios de Aceite

- [ ] BullMQ instalado e configurado (`bullmq`, `ioredis`)
- [ ] Conexão com Redis (Upstash) funcional via `REDIS_URL`
- [ ] 4 filas criadas: `document-processing` (alta prioridade), `petition-generation` (média), `batch-import` (baixa), `index-sync` (cron)
- [ ] Workers placeholder criados para cada fila (log de recebimento)
- [ ] Retry com backoff exponencial configurado por fila
- [ ] Dead letter queue configurada para jobs que falharam após retries
- [ ] Health check de conexão Redis funcional
- [ ] BullBoard (dashboard de monitoramento) acessível em `/api/queues` (admin only)
- [ ] Variáveis de ambiente: `REDIS_URL`, `REDIS_TOKEN`
- [ ] Graceful shutdown dos workers

## Notas de Implementacao

- Upstash Redis requer TLS — usar `ioredis` com `tls: {}` e `maxRetriesPerRequest: null` para BullMQ
- Configuração de retry por fila:
  - `document-processing`: 3 attempts, backoff exponencial 5s
  - `petition-generation`: 2 attempts, backoff exponencial 10s
  - `batch-import`: 1 attempt, prioridade 10 (baixa)
  - `index-sync`: cron `0 6 * * *` (diário às 6am)
- Workers devem rodar em processo separado em produção (Railway), mas em dev podem rodar no mesmo processo
- BullBoard deve ser protegido — apenas admin pode acessar `/api/queues`
- Dead Letter Queue (DLQ): jobs que falharam após max retries vão para fila `{name}-dlq`
- Connection pooling: reutilizar conexão Redis entre filas
